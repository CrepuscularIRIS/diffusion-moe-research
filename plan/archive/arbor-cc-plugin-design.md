# Design: `arbor-cc` — a Claude Code plugin that is a disciplined `/goal` + Arbor cycle

Generated by /office-hours on 2026-06-25 · Status: DRAFT (approved approach: B) · Mode: Builder/Tooling
Companion: `stophook.md` (the Stop-hook mechanism), `arbor-peer-review-refactor-plan.md`,
`arbor-codex-routing.md`, `research-goal.md`, `ai-research-conduct-principles.md`.

## Problem

Bare `/goal` failed in practice: an 8-hour CPU hang (no cycle/time cap), unverified
"done" claims (no evidence gate), cycles run without acceptance criteria actually
extracted from `research-goal.md`, and an opaque blackbox the user couldn't steer or
watch. We want **Arbor's discipline (idea tree, phases, gates, isolation, review)
driven entirely by Claude Code (Opus)**, packaged as one plugin, with `/goal`-level
autonomy available but **controllable** and **observable** — and **no API key**.

## Decision: Approach B (keyless plugin + arbor read-only viz)

Driver = Claude Code (Opus). State/tree = `arbor mcp`. Phase logic = the 11
`arbor-agent-*` skills + `arbor-peer-review-gate` + `diffusion-moe-research-profile`.
Loop control = a fixed **Stop-hook gate + state files** (the `stophook.md` pattern).
Visibility/output = `arbor web` / `arbor report` (read-only, no key).

### Premises (confirmed)
1. `/goal` = a session-scoped prompt Stop hook. A **fixed command Stop hook + dynamic
   state file** reproduces ~90% of it and is *more* controllable (`stophook.md` §1-8). ✓
2. Arbor's structure is already in CC keyless: idea tree via `arbor mcp`, phase logic
   via the installed skills. The native CLI runtime only adds an LLM session. ✓
3. **`arbor web SESSION` and `arbor report SESSION` need no key and read
   `.arbor/sessions/<run>/` — the same path the MCP server writes
   (`.arbor/sessions/diffusion-moe/.coordinator/`).** So live monitoring of a
   keyless CC-driven run works out of the box. ✓ (verified 2026-06-25)
4. Rejected (Option 2): an `arbor run` + Kimi runtime would put the **weakest model in
   the coordinator seat** (IDEATE/SELECT/DECIDE is the heavy reasoning), and the
   arbor↔CC hook bridge does not exist and would be fragile. ✗

## Architecture — plugin `arbor-cc`

```
arbor-cc/                      (a Claude Code plugin)
├─ .claude-plugin/plugin.json  manifest
├─ hooks/
│   └─ arbor-stop.mjs          the deterministic Stop gate (command hook)
├─ commands/  (or skills/)
│   ├─ arbor-init.md           read research-goal.md → QUEUE + cycle-state + tree
│   ├─ arbor-cycle.md          run ONE cycle to the next gate, then stop
│   ├─ arbor-dispatch.md       executor in worktree (+ science-protocol inject)
│   ├─ arbor-decide.md         BACKPROP + Codex DECIDE + milestone peer-review
│   ├─ arbor-review.md         peer-review panel on demand
│   ├─ arbor-status.md         show tree/QUEUE/RUNLOG/debt + launch `arbor web`
│   ├─ arbor-report.md         `arbor report` → REPORT.md
│   └─ arbor-clear.md          active=false
├─ .mcp.json                   declares the `arbor mcp` server
└─ settings (Stop hook registration)
```
Bundled-by-reference (not duplicated): `arbor-agent-*`, `arbor-peer-review-gate`,
`diffusion-moe-research-profile` skills; Codex + Playwright stay as the existing
3-engine routing.

### Per-run state files (the `stophook.md` "dynamic state file")
Under `.arbor/sessions/<run>/` (so `arbor web` sees them):
- `cycle-state.json` — `{active, mode: "review"|"auto", goal, maxCycles, cycle, started}`.
- `QUEUE.md` — the cycle backlog. **Each item = one cycle's GOAL + a pre-declared
  ACCEPTANCE/falsifier**, extracted from `research-goal.md` §4 (H1-H8) and §10 at
  `/arbor-init`. This is the thing bare `/goal` never built.
- `RUNLOG.md` — evidence: what ran, exact commands, exit codes, git status, scores,
  per-job expected-vs-actual wall-clock.
- `BLOCKED.md` — blockers needing a human.
- `review_debt.md` — open Reject→Accept items from the peer-review gate (per the
  refactor plan); OBSERVE consumes it with priority over new ideas.

### The Stop gate — `arbor-stop.mjs` (command hook, deterministic)
Fires every turn (Stop has no matcher); guards via `cycle-state.json`. Logic:
- If no `cycle-state.json` or `active=false` → `exit 0` (no-op).
- `stop_hook_active === true` → `exit 0` (never infinite-block; `stophook.md` §4).
- **maxCycles cap** → if `cycle > maxCycles`, force `active=false`, block with
  "stop new work; summarize state/blockers/changed-files/next slice." (kills the 8h-hang
  class of failure mechanically.)
- **Evidence gate** → before any stop, require a fresh `RUNLOG.md` (commands + exit
  codes + git status). Missing → block with "update RUNLOG with evidence first."
- **Process-hygiene gate** → if RUNLOG declares a background job with no recorded
  exit/kill and it has exceeded its declared budget, block with "kill the overrunning
  job (`kill -9`) and record it" (operationalizes the kill-on-timeout lesson).
- **Mode branch:**
  - `review` (default) → once the current cycle reached its declared gate and RUNLOG is
    fresh, **allow stop** (the user reviews, then runs the next `/arbor-*` command). The
    gate enforces *clean* stops, it does NOT force continuous running.
  - `auto` → if the cycle's acceptance is unmet, **block** and feed the next cycle step
    (this is the disciplined `/goal` — same autonomy, but now capped + evidence-gated).

So the same hook gives **two modes from one mechanism**: disciplined-autonomous
(`auto`, = a safe `/goal`) and human-paced (`review`, = the control the user now wants).

### Slash commands (one cycle, mapped to the Arbor phases)
- `/arbor-init <goal|path>` — **read `research-goal.md`**; extract H1-H8 + §10 success
  criteria into `QUEUE.md` as concrete per-cycle GOAL+ACCEPTANCE; create
  `cycle-state.json`; init the MCP idea tree + metadata; load the profile skill.
- `/arbor-cycle` — OBSERVE (tree + `review_debt` first + RUNLOG) → IDEATE (probe block,
  `TreeView(constraints)` → propose nodes) → SELECT (Codex `/codex:rescue`) → present
  the selected experiment **+ its pre-declared falsifier** → stop for user OK (review
  mode).
- `/arbor-dispatch` — executor subagent in a git worktree, science-protocol injection,
  **declare expected wall-clock**; on return, Codex reviews the diff.
- `/arbor-decide` — BACKPROP + Codex DECIDE (merge/prune, "gain from mechanism or
  artifact?"); if a depth-1 Direction hit a milestone (refactor-plan §3) → load
  `arbor-peer-review-gate` → append AC action list to `review_debt.md`.
- `/arbor-review` — run the 5+AC panel on demand.
- `/arbor-status` — print tree (`tree_view`) + QUEUE + RUNLOG + open review_debt; launch
  `arbor web <run>` for the live browser monitor.
- `/arbor-report` — `arbor report <run>` → REPORT.md (unresolved FATAL/MAJOR → caveats).
- `/arbor-clear` — `active=false`.

## How it fixes each failure we hit
| Past failure | Fix in `arbor-cc` |
|---|---|
| 8h CPU hang | `maxCycles` cap + process-hygiene gate in the Stop hook |
| Unverified "done" claims | RUNLOG evidence gate (commands + exit codes) blocks stop |
| `research-goal.md` never read for acceptance | `/arbor-init` extracts H1-H8/§10 → QUEUE per-cycle GOAL+验收 |
| Opaque / can't steer | `review` mode (user paces each `/arbor-*` step) |
| Can't watch | `arbor web <run>` live tree monitor (same `.arbor` format) |
| Lost work on stop | every stop is a clean checkpoint (state + RUNLOG persisted) |
| No publishability check | `/arbor-decide` milestone → peer-review gate → review_debt |

## Success criteria
1. `/arbor-init` on `research-goal.md` produces a QUEUE where every item has a concrete
   GOAL + a killable ACCEPTANCE (maps to an H1-H8 falsifier or a §10 criterion).
2. In `review` mode, one `/arbor-cycle` runs exactly to the next gate and stops; the
   user can `arbor web` the tree live.
3. In `auto` mode, the loop never exceeds `maxCycles`, never stops without a fresh
   RUNLOG, and never lets a declared job overrun its budget without a kill.
4. A Direction milestone triggers the peer-review gate and writes `review_debt.md`,
   which the next OBSERVE consumes first.
5. Installs as one plugin; no API key; Opus is always the coordinator.

## Open questions (verify during build)
- Exact MCP tree JSON path under `.coordinator/` vs what `arbor web` renders (dir
  matches; confirm field-level compatibility on first `arbor web` run).
- Stop hook can *block-with-instruction* to kill overrunning jobs, but cannot kill them
  itself — confirm the "agent kills on next turn" loop is tight enough, or add a
  `PreToolUse` budget reminder.
- `disable-model-invocation` on the commands so they only fire on explicit user/`auto`
  invocation, not model whim.

## What I noticed
- You diagnosed the real failure precisely: "直接用 goal 是不可行的" — not because CC
  can't loop, but because the loop had no gate. `stophook.md` is you having already
  found the fix; this design just packages it.
- You proposed the Kimi-runtime option but framed it as "不知道是否可行" — the honest
  doubt was correct: the weak point is exactly Kimi-as-coordinator, which you sensed.
