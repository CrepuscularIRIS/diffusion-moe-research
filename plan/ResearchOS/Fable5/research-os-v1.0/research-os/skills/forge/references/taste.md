# Taste — the model behind research-os

## 1. What taste is
**Taste is a prior over expected-information-per-unit-effort, operating BEFORE evidence exists.**
Verification is posterior — it judges results already paid for. Taste is prior — it decides what is worth
paying for. This is why no gate stack can substitute for it: by the time a gate fires, the money is spent.
A system that is 90% filter produces well-audited mediocrity.

Two modes, and the difference is everything:
- **Evaluative taste** — *"is this good?"* Rules can encode it. It can only subtract.
- **Generative taste** — *"what would make this good?"* The mode that produced ResNet, DSpark, MAE. It adds.

Generative taste is the objective function of `/prospect` and `/forge`; taste filters only once, at
`/adversary`'s worth check.

## 2. The four components
- **Problem taste** — load-bearing? (if solved, a decision/method/belief changes — "nothing changes" = not
  a problem). Hidden by the field's default framing? (第一性原理 / omitted-structure: the best problems are
  invisible from inside the default parameterization). **Now-possible?** — demand a WHY-NOW or a WHY-EMPTY.
- **Mechanism taste** — one mechanism doing the work, stateable in one sentence before running. **Elegance =
  compression**: a pile of compensations patching a wrong object is debt; a single natural object that makes
  them unnecessary is compression — the fix and the diagnosis are one object (自解释).
- **Evidence taste** — the cheapest experiment with the highest expected information. **Kill before
  confirm. One variable per probe. Surprise-symmetric**: every outcome teaches; an experiment whose failure
  would be dismissed and whose success was expected is a wasted GPU-day.
- **Timing taste** — why now: a new capability, a new failure at scale, a fresh contradiction.

## 3. Surprise accounting — the unit of progress
**The unit of progress is a SURPRISE: an observation that changed the plan.** Not a token, not a banked
artifact, not a passed gate. Surprise cannot be farmed by orderly retreat — an expected failure changes no
plan. Taste maximizes expected plan-change per GPU-hour.

## 4. The He-bar (type-relative; generative first, filter last)
少 few parts · 准 hits the load-bearing variable · 可复述 survives compression to one sentence ·
可证伪 a pre-stated result would kill it · 可生成 spawns the next questions · 可命名 becomes vocabulary ·
可迁移 survives off its home benchmark · 自解释 the mechanism explains the failure it fixes.

The bar is **TYPE-relative** (`research-types.md`): an improvement needs 准+少+可证伪, not 可命名; a novelty
claim needs the full bar. Use it **generatively first** — five real minutes on *"what would make this
beautiful?"* before building — and as a filter only at the claim boundary.

## 5. The five failure modes (the cut's targeting function — each observed in practice)
1. **Novelty-worship** — unoccupied ≠ valuable. Occupancy is a cost signal, never a value signal.
2. **Rigor-worship** — verified ≠ important. A perfect null nobody needed is still zero.
3. **Ceremony** — process artifacts treated as outputs. If recent artifacts are mostly about the workflow,
   the workflow has become the product.
4. **Menu-thinking** — enumerating candidates up front makes a queue, and queues exhaust. A live programme
   regenerates its backlog from each failure.
5. **Gate-shaped work** — a gate-heavy system selects for the most gateable work, which is
   measurement/certificates. If outputs drift to certificates when the goal says design, the incentives
   are broken, not the ideas.

## 6. Taste and the one invariant
Taste-KILL is self-administrable (killing your own toy cannot be gamed upward). Taste-PASS toward
"contribution" is not: it runs on an independent substrate and terminates in a human. Generative taste is
bold; promotional taste is conservative. Both are taste; only their authority differs.
