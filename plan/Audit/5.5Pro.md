可以优雅解决，而且**最优雅的一步不是加更多搜索、更多 Agent、更多 Pro 报告，而是把 Research-OS 的核心对象从「Mechanism Candidate」改成「Failure Signature / Problem State」**。

也就是：系统不再先问“我能设计什么机制？”，而是先问：

> **有没有一个经过强基座、强提示、公平 LoRA、数据规模扩展、seed/OOD 检查之后仍然稳定失败的局部失败簇？**

只有这个存在，才允许进入 mechanism design。

---

## 真正缺的那一层

你现在的系统其实已经建模了很多东西：

* taste：什么像贡献，什么像 incremental；
* rigor：公平 baseline、seed、消融、negative control；
* autopsy：失败后复盘；
* region-close：避免在死区继续浪费时间；
* Pro / Codex / Opus 分工：生成、审计、执行、裁决。

但它缺了一个最关键的中间对象：

> **局部问题模型。**

也就是 Failure Atlas。

现在每个负结果大概被记录成：

> 这个机制死了。
> 这个 benchmark 可能饱和。
> 这个方向 region-close。

但研究者真正积累的是：

> 哪一类样本死？
> 哪个 baseline 为什么赢？
> LoRA 解决的是表层映射还是不变量？
> metric 有没有掩盖真正失败？
> 哪个 OOD split 才是真断裂？
> 哪些样本是 deceptive cases？
> 哪些失败跨 seed、跨 prompt、跨模型稳定？
> 哪些 failure 是机制空间，哪些只是数据规模问题？

这个没有被充分建模进去，所以系统会“很会杀”，但不会“越杀越懂”。

---

## 一步修改：把主循环入口改成 Necessity Gate + Failure Atlas

不是重写所有 command。只改一个总规则：

Research-OS must not generate mechanisms for a benchmark or topic. It may only generate mechanisms for a verified failure signature: a stable cluster of examples where the frozen base model, tuned prompting, retrieval/tooling if applicable, fair plain LoRA, reasonable data-scale expansion, seed checks, and declared OOD/compositional/robustness splits still fail under a specific invariant. Every cycle must either strengthen, weaken, or refine the Failure Atlas for that signature; lateral movement and region-close are forbidden until the Atlas shows that no stable residual failure mode remains.

这就是最小补丁。

它把系统从：

> Survey → idea → mechanism → probe → kill → lateral

改成：

> Human/AI bet → Necessity Gate → Failure Atlas → mechanism → differential test → atlas update → deepen or close

---

## 为什么这一步能解决你现在的问题

因为你最近连续死掉的机制，本质不是“机制生成不够聪明”，而是它们进入机制设计之前没有先证明：

> **这里真的需要机制，而不是普通 LoRA、prompt、attention pruning、VCD、larger frozen VLM 或数据规模就能解决。**

所以 `Necessity Gate` 是第一道硬门。

它要求在任何 mechanism 之前先跑：

1. frozen base；
2. tuned prompting；
3. 任务适用时的 retrieval/tool baseline；
4. plain LoRA / QLoRA；
5. 数据规模小 sweep；
6. seed variance；
7. OOD / compositional split；
8. metric sanity；
9. negative control。

如果 LoRA 解决了，那不是“机制失败”，而是：

> **BENCHMARK_OCCUPIED：这个 benchmark 不适合做 mechanism discovery。**

如果 LoRA 也解决不了，并且失败集中在稳定样本簇上，才进入：

> **MECHANISM-WORTHY FAILURE SIGNATURE。**

---

## 搜更多综述、GPT-5.5 Pro 报告、Deep Research，有没有用？

有用，但不能放在最前面当“方向生成器”。

它们应该降级成 **Atlas Enrichment Tool**，也就是在一个失败区域已经被锚定之后，用来补充四类信息：

| 信息源                                        | 正确用途                                   | 错误用途                   |
| ------------------------------------------ | -------------------------------------- | ---------------------- |
| 综述论文                                       | 找长期公认的难轴、常用数据集、评价问题                    | 直接从“gap 列表”挑论文题        |
| GPT-5.5 Pro / Deep Research                | 做 occupancy、baseline、机制类比、反方审计         | 让它自动生成下一个伟大 idea       |
| 最新论文 / repo / leaderboard                  | 找最强 tuned baseline，防止撞车                | 只看 abstract 判断 novelty |
| 社区评价 / GitHub issue / benchmark discussion | 找 hidden baseline、数据泄漏、metric bug、实现陷阱 | 把社区观点当证据               |

所以不是“不要搜”，而是搜索必须服务于一个已经存在的 Failure Signature。

搜索输出不能是“我读了 30 篇论文”，而必须是 Failure Atlas 的行：

```text
Failure cluster:
Baseline that fails:
Baseline that unexpectedly works:
Metric pathology:
Known prior art:
Likely occupancy risk:
Probe needed:
What would kill this problem:
```

这才叫 depth。

---

## 你说的“研究者会做记录”，对，就是这里没建模好

研究者的 depth 不是“读了很多论文”。

真正的 depth 是：

> 在同一个对象上反复失败，然后逐渐知道哪里是真的硬，哪里只是错觉，哪里是 benchmark 假象，哪里是 baseline 已经占掉的空间。

你现在的系统有 Negative Result Memory，但它更像全局墓碑：

> 这个方向死了。
> 那个机制死了。
> 这个 region 关闭。

它还不够像局部地形图：

> relation-precondition 在 4-step 以上坍塌；
> content F1 还能学，但 dependency F1 不随 LoRA 同步增长；
> gold-step recall 是 substrate gate；
> occlusion/blocker recall 是瓶颈；
> tuned prompting 提升 parse，不提升 dependency；
> plain LoRA 若只提升 surface order 而不提升 precondition edge，则机制空间还在。

你最近的 SpatiaLQA 其实已经更接近正确形态了，因为它不是空泛地说“空间推理有 gap”，而是已经有了：

> content F1 0.427，但 precondition F1 0.0555，parse 0.977，人类 92，说明是真 dependency gap。

这就是 Failure Signature 的雏形。

下一步不是横跳，而是把这个失败簇继续压深：plain LoRA 到底能不能补 precondition？大模型 frozen 到底是 perception substrate 问题还是 dependency induction 问题？失败是否集中在 occlusion、blocker、long-step、support relation？这些才是 atlas。

---

## 最终结论

你的自动化科研不是“没救”，也不是“需要更多文献”。它的问题是：

> **它把 Research Depth 建模成了更多候选方向 + 更强审计，而不是同一失败区域内的局部地图积累。**

最优雅的一步就是：

> **把 `/prospect` 或 `/forge` 的入口条件改成：没有 LoRA-resistant Failure Signature，就不允许生成 mechanism。每轮必须更新 Failure Atlas，而不是只产出 kill / region-close。**

这样以后 Pro、Deep Research、综述论文、社区评价都还有用，但它们不再是方向源头，而是为一个已锚定失败区域提供 baseline、occupancy、mechanism analogy 和 hidden-trap 信息。

你不需要把 Research-OS 推倒重来。
你只需要把系统的主语从：

> “我要找一个新机制”

改成：

> “我要解释并修复一个公平 baseline 仍然解决不了的稳定失败簇。”

这一步改完，整个系统的方向会立刻从“广度生成器”变成“深度压缩器”。
