# Modeling Taste Atlas 的 Taste Operators 银行

## 抽取口径与总判定

这次不是按论文、也不是按领域复述，而是把你上传的六份报告里的“旧对象 → 新对象”案例重新压缩成**可迁移的研究动作**。我的合并原则是：只有当一个模式同时满足“移除旧复杂性”“带来旧对象做不到的差异预测”“能设计 cheap probe”“能被明确反证”这四条时，才进入 operator bank；凡是仅仅换 backbone、把东西统一成 token、加模块、换 benchmark、扩大数据或把 carrier 名词当 operator 的，一律不进 KEEP。基于这套过滤，我把重复项合并后得到 **32 个 KEEP operators、7 个 ARCHIVE operators、9 个 KILL pseudo-operators**。这个切分与几份报告里反复出现的主轴一致：density 被改写为 ratio/score/velocity/transport，scene 被改写为 set/object/graph/field，state 被改写为 belief/context/latent plan，trajectory 被改写为 distribution/chunk/skill，simulation gap 被改写为 residual/context/ensemble/invariant subspace。 fileciteturn0file0 fileciteturn0file1 fileciteturn0file2 fileciteturn0file3 fileciteturn0file4 fileciteturn0file5

## KEEP / ARCHIVE / KILL 清单

先给结论：**KEEP 里的 32 个 operator 已经足够直接生成新研究候选**；ARCHIVE 里的概念值得保留，但在你当前 atlas 中更适合作为“候选源 episode”或“待审计标签”；KILL 则应当在 operators.md 中明确排除，以免把 buzzword 和真正的建模动作混在一起。这个切分基本对应你上传报告里的研究口径：要找的是中介对象和可杀死实验，而不是漂亮术语。 fileciteturn0file1 fileciteturn0file2 fileciteturn0file3

| operator_name | 归档原因 | 当前状态判断 |
|---|---|---|
| prediction-setification | cheap probe 很清楚，但在上传报告中的历史 episode 主要集中在 conformal prediction 一条线，跨域重复还不够厚 | 概念强，等更多 VLA / MLLM / runtime case 再升 KEEP |

| low-rank-deltaization | LoRA 很强，但跨域重复目前多集中在参数适配与少数低秩修正案例 | 适合做 PEFT 分支，不宜和 belief/field/operator 同层级并列 |
| prompt-conditioning externalization | 从 full fine-tune 到 soft prompt/prefix 的动作有用，但差异预测常常依赖具体底模接口 | 候选，需更多 runtime/VLA 明确案例 |

| worst-groupization | Group DRO 很实用，但它更像目标重加权而非“换中介对象”的强母式 | 作为 invariantization 的邻近概念保留 |

**C. KILL**

| pseudo_operator | 为什么应杀 |
|---|---|
| tokenization | “统一成 token”本身不是 operator；除非 token 化显式改变了可预测/可控制对象，否则只是接口统一 |
| backbone swap | 换 Transformer、Conv、Mamba、ViT 不是对象转移 |
| bigger-data scaling | 更大数据只改变统计覆盖，不自动改变建模对象 |
| bigger-model scaling | 更大参数量不是 operator |
| module adding | “加 memory / add graph / add adapter”如果没有对象切换，只是工程堆叠 |
| benchmarkization | 新 benchmark 会暴露问题，但不是研究动作本身 |
| field / graph / latent / energy / operator 这些 carrier 名词 | 它们是数学载体，不是 taste operator；真正的 operator 必须说清“为什么换它” |
| multimodal concatenation | 把模态拼起来不是 operator；没有新的 sufficient object 就只是拼接 |
| end-to-end unification | “统一模型/统一接口/统一 token 空间”若没有新的 differential prediction，应直接判伪 |

我对这 32 个 KEEP 的总体判断是：**最强的一批不是在“拟合更复杂对象”，而是在“发明一个新的中介对象，把复杂性换到另一个可控空间里”**。如果你要把 atlas 真正变成下一篇论文的生成器，那么先进 bank 的，优先应是 beliefification、object-slotting、fieldization、operator-lifting、residualization、gap-contextification、memory-kernelization、transportization、trajectory-chunking 这一类——因为它们既跨域重复，又有明确 cheap probe，还能在大规模训练前被快速杀掉。 fileciteturn0file0 fileciteturn0file2 fileciteturn0file3 fileciteturn0file5
