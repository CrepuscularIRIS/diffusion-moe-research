## 总体判断

你的核心判断是对的，而且问题比“对 Taste 的定义不够准确”更严重：

**这套工作流建模的主体不是 AI 算法工程师，而是一个科研项目经理、程序委员会和实验审计系统的混合体。**

它实际上把一个完整的算法研究者拆成了几个人：

| 当前角色分工                 | 实际承担的能力  |
| ---------------------- | -------- |
| Human                  | 提供 Taste |
| External brain / panel | 提出数学与模型  |
| Conductor              | 编排、调和和简化 |
| Executor               | 编码与运行    |
| Jury / adversary       | 排序、评判和批准 |

问题在于：**没有任何一个主体持续拥有“问题—算法—代码—实验—残差诊断”的完整闭环。**

而 AI 算法研究最核心的能力恰恰来自这个闭环。算法直觉不是在外部定义好 Taste、生成数学、再把代码交给执行器之后形成的；它来自同一个主体反复观察：

* 某种误差究竟是表示问题、优化问题、数据问题还是评价问题；
* 某个公式落到代码后，梯度路径和数值行为是否与预期一致；
* 某个模块带来的收益究竟来自所宣称的机制，还是容量、算力或实现差异；
* 实验残差应该怎样反向修改算法。

所以，当前工作流最根本的问题可以概括为：

> 它试图通过建模“优秀研究是怎样发生的”来制造优秀研究，而不是让模型直接进入算法工程师的工作状态。

这属于一次不必要的元层建模。

## 一、Taste 的建模发生了类别错误

当前 `taste.md` 将 Taste 定义为一种：

> evidence 之前，对单位投入预期信息量的先验。

这个定义不是毫无价值，但它定义的主要是：

**实验选择策略或资源分配策略，而不是算法 Taste。**

至少应当分开三个对象：

1. **算法 Taste**：什么瓶颈值得攻击，什么算法变化最有杠杆。
2. **实验策略**：哪个实验具有最高的信息增益／成本比。
3. **证据策略**：已有结果足以支持多强的结论。

当前系统把这三者压进了一个 Taste 概念，于是会自然偏向：

* 便宜的小实验；
* 容易证伪的候选；
* 容易向 jury 解释的机制；
* 能产生 surprise 的方向；
* 易于流程化审计的工作。

但高水平 AI 研究中的重要算法赌注，未必在早期具有最高的信息增益／成本比。有些机制必须达到一定数据规模、模型规模或训练时长后才会显现；有些组件必须协同变化，不能永远坚持“一次只改一个变量”。

因此：

> `expected information / effort` 可以保留，但应该叫 experiment policy，不应叫 Taste。

更准确的算法 Taste 可以定义成：

> **从真实系统的误差结构、容量边界或优化病态中，识别最承重的瓶颈，并把它压缩成一个可执行、可诊断、可扩展的算法改动。**

这里的关键词不是 surprise，而是：

* 承重瓶颈；
* 算法杠杆；
* 可执行性；
* 可诊断性；
* 跨尺度稳定性。

## 二、最明显的结构性错误：禁止主导者亲自建模

`skills/conductor/SKILL.md` 第 63–69 行附近明确规定：

> conductor never authors mathematics itself

`skills/forge/SKILL.md` 也重复了数学建模应当委派给 panel 或 external brain。

这是整个系统最需要优先删除的规则。

对于一般项目管理，主协调者不亲自完成每项专业工作可以成立；但对于 AI 算法研究，这会切断五个高度耦合的对象：

[
\text{问题表述}
\rightarrow
\text{数学对象}
\rightarrow
\text{计算图}
\rightarrow
\text{代码行为}
\rightarrow
\text{实验残差}
]

高水平算法工程师需要知道：

* 为什么选择这个表示，而不是另一个表示；
* 这个目标函数究竟对哪些参数施加了什么压力；
* credit assignment 如何穿过计算图；
* 哪部分梯度会饱和、消失、爆炸或被噪声淹没；
* 训练阶段与推理阶段是否存在结构失配；
* 理论上“等价”的表达在有限精度、分布式训练和具体 kernel 下是否仍然等价。

这些不能稳定地由一个主体提出数学、另一个主体调和、第三个主体编码、第四个主体诊断来完成。

协作当然有价值，但正确关系应是：

> **算法负责人亲自拥有 working model；外部模型提供替代方案；评审者负责攻击假设；执行器帮助实现，但不替代算法所有权。**

换句话说：

**可以委派计算，不能委派理解。**

## 三、把证据独立性错误地扩展到了创意选择

这套工作流有一个合理原则：

> proposer 不能独自给自己的结论发放可信度。

在论文结论、统计证据、基线公平性和运行真实性上，这个原则正确。

但当前 Prospect 和 MoA routing 又把它扩展成：

* 生成者不能评价自己的候选；
* Taste 排序必须由跨 family jury 完成；
* jury 对候选方向作 program-chair 式裁决；
* 候选还要先满足 GOOD/BAD Taste anchors；
* 必须经过 refinement demand 和并行 micro-pilot 才能退出。

这相当于把：

> “不能给自己的证据盖章”

错误推导成了：

> “不能依赖自己的算法直觉选择研究方向”。

两者不一样。

创意阶段需要的是高度一致、甚至带有个人偏见的机制直觉。很多新方向最初都不容易被标准化 reviewer 语言表达。jury 往往会隐式优化：

* 是否容易解释；
* 是否符合已有论文范式；
* 是否像一个顶会 story；
* 是否能迅速形成完整实验矩阵；
* 是否符合多数评审者的先验。

这会提高可评审性，却不一定提高原创性。

正确的边界应当是：

> **创造权集中，证据权分离。**

算法负责人可以自己提出、选择甚至坚持一个非共识算法方向；但一旦要声称它有效、优于基线或具有普遍性，就必须接受独立、严格、artifact-backed 的验证。

## 四、Operator Bank 与 Taste Anchors 过于间接

当前系统试图从历史突破中提取抽象 operator，例如：

* 某种跨领域重组；
* 某种问题变换；
* 某种结构替换；
* 某种 mechanism pattern。

这类资料可以作为可选启发，但不应成为主导生成过程的机制。

原因是，事后对论文的“突破机制”进行抽象具有很强的非唯一性。同一篇论文可以被解释成：

* 表示创新；
* 优化重参数化；
* 归纳偏置变化；
* 系统扩展性创新；
* 数据规模变化后的结构简化。

事后总结出的 operator 很容易形成一种研究神话：它在回顾时非常清楚，在生成新算法时却缺少约束。

更有效的研究记忆单位不是抽象 operator，而是：

```text
Context:
Observed bottleneck:
Exact algorithmic intervention:
Expected behavioral signature:
Observed behavioral signature:
Failure boundary:
Scaling behavior:
```

也就是把记忆组织成：

> **故障现象 → 算法改动 → 可观察签名 → 适用边界**

这比“某篇突破使用了什么抽象创新算子”更接近算法工程师真实可调用的经验。

Taste Anchors 也有类似问题。大量 GOOD/BAD anchors 会让模型学习“什么看起来像好论文”，而不是“当前系统中什么算法变化能直接攻击瓶颈”。

因此，问题不只是字数多，而是：

> **工作流把注意力从 artifact 和 computation graph 拉向了 protocol bookkeeping。**

# 我对 AI 算法工程师的理解

这里的“AI 算法工程师”不是狭义的调参工程师、模型复现工程师或代码实现人员，而是：

> **在数学对象、学习算法、计算系统和实验证据之间保持闭环所有权的研究型工程师。**

他至少需要完成四种闭环。

## 1. 数学闭环

能把模糊直觉写成明确的：

* 输入、输出和状态；
* 表示对象；
* 目标函数；
* 更新规则；
* 推理过程；
* 约束和假设。

不一定要求复杂数学，但必须精确到能够实现，不能停留在概念名词。

## 2. 计算闭环

知道数学表达进入真实系统后发生什么：

* FLOPs、显存和通信；
* 并行方式；
* 数值稳定性；
* kernel 和张量布局；
* 训练与推理差异；
* 延迟、吞吐和可扩展性。

在 AI 中，“算法”和“系统”经常无法严格分开。一个理论上漂亮但无法稳定训练、无法扩展或无法高效推理的机制，并不是完整的算法贡献。

## 3. 实证闭环

在实验前就能说出：

* 如果机制正确，哪些曲线会变化；
* 哪种错误类型会减少；
* 哪些层、token 或样本上会先出现变化；
* 哪个 ablation 应当摧毁收益；
* 哪些结果说明收益来自容量而不是机制。

也就是说，算法不是“改完看分数”，而是必须携带预测。

## 4. 残差闭环

实验失败后，不只是输出“假设被否定”，而是能区分：

* 假设错误；
* 实现错误；
* 优化未进入有效区域；
* 测试尺度太小；
* 数据分布不支持；
* 指标对机制不敏感；
* 多组件之间发生了 co-adaptation。

然后把残差转化为下一次算法修改。

真正的 Taste 就在这种循环里形成，而不是来自静态 rubric。

# 更准确的 AI 算法 Taste

我会把它压缩成七种直觉，但这些应当是角色先验，不应再次做成七道硬 gate。

### 1. Bottleneck sense

能判断当前真正限制系统的是：

* 表示能力；
* credit assignment；
* 优化条件数；
* 数据效率；
* 长程依赖；
* 搜索复杂度；
* 记忆与上下文；
* 推理预算；
* 评价失真。

### 2. Object sense

知道“应该对什么对象建模”。

许多重要算法进步首先改变的不是模型规模，而是基本对象：

* layer 学 residual 还是完整映射；
* 图像被视为局部卷积网格还是 patch sequence；
* 参数更新发生在全空间还是低秩子空间；
* 学习目标是直接预测标签、噪声、score，还是未来适应能力。

### 3. Computational-path sense

能看见信息和梯度怎样流动：

* 哪些路径太长；
* 哪些依赖无法并行；
* 哪些模块造成瓶颈；
* 哪些跳连、归一化或重参数化能够改变优化几何。

### 4. Minimal-mechanism sense

追求的不是最少代码，而是：

> 最少的新假设、最短的因果链、最容易被单独证伪的承重机制。

### 5. Implementation sense

在提出算法时就考虑真实实现，而不是先抽象、再把“工程问题”交给另一个角色。

### 6. Signature sense

每个算法假设都应对应一组可观察行为，而不只是最终 benchmark 数字。

### 7. Scaling sense

知道局部收益是否可能：

* 随数据规模增强；
* 随模型规模消失；
* 受硬件限制；
* 只在小数据下成立；
* 需要达到某种临界规模才能显现。

# 对代表性顶会研究者和论文风格的直观理解

这里不是给研究者做绝对排名，而是从代表性高影响论文中归纳算法 Taste。

## Kaiming He：找到承重参数化，然后删除不必要结构

ResNet 的关键不是构造一套复杂研究流程，而是把“直接学习目标映射”改成“学习 residual，再加 identity path”。这是一个非常具体的参数化变化，它直接改变优化路径，同时可以通过深度扩展和训练行为进行验证。MAE 同样把视觉预训练压缩成高比例 masking、只编码可见 patch、轻量 decoder 重建的非对称结构；它同时处理了表示学习和计算效率。我的直观归纳是：他的 Taste 经常表现为**识别现有计算结构中最承重的阻力，然后用一个非常清晰的结构变化解除阻力**。([CVF Open Access][1])

这不是“不建模”，而是模型极其直接：公式、计算图和实现几乎是同一个东西。

## Transformer 团队：重新设计计算图，而不是修补旧范式

Transformer 的核心判断不是再给循环网络增加若干机制，而是直接去掉 recurrence 和 convolution，把序列依赖主要放到 attention 中。这既改变了信息交互方式，也改变了并行性和训练计算路径。其 Taste 在于看见：旧结构的顺序计算并不是任务本身不可避免的要求。([NeurIPS Proceedings][2])

这种研究不是从“有哪些创新 operator”出发，而是从：

> 当前计算图究竟为什么低效、为什么难以传播信息？

出发。

## ViT 团队：知道何时可以删除传统归纳偏置

ViT 把图像拆成 patch sequence，直接应用标准 Transformer。关键 Taste 不只是“把 NLP 方法迁移到视觉”，而是判断：当数据和规模条件发生变化后，卷积中的某些领域特定归纳偏置是否仍然是必要的。论文显示在足够规模的预训练下，纯 Transformer 能取得很强的视觉性能。([arXiv][3])

这里体现的是一种非常重要的算法工程师直觉：

> 不只知道应该增加什么，还知道环境变化后什么可以删除。

## LoRA：从参数更新的几何结构切入

LoRA 没有重新设计一套庞大的微调框架，而是假设任务适配所需的权重变化具有低内在秩，在冻结预训练参数的同时注入可训练低秩矩阵。它同时改变训练参数子空间、显存需求和部署方式。([arXiv][4])

这是一种典型的强算法 Taste：

1. 找到成本真正来自哪里；
2. 对更新空间提出结构假设；
3. 用一个很小的数学改动解决系统级问题；
4. 让假设能够通过 rank、参数量和性能实验直接验证。

## MAML：直接重写优化目标

MAML 的贡献不是增加一个复杂的任务管理层，而是把“能够快速适应新任务”直接写进训练目标：训练出的参数应当在少量梯度更新后获得良好表现。它把一个模糊能力要求变成了可执行的优化算法。([Proceedings of Machine Learning Research][5])

这说明顶级 AI 论文可以非常抽象；关键是这种抽象必须直接落到 objective 和 update rule，而不是停留在研究流程或概念分类上。

## Soft Actor-Critic：从学习动力学与目标函数解决不稳定性

SAC 用最大熵目标和 off-policy actor-critic 结构处理强化学习中的样本效率和训练稳定性问题。这里的算法 Taste 是：不把不稳定性仅仅当作工程噪声，而是从目标函数和数据使用方式上直接改变学习动力学。([Proceedings of Machine Learning Research][6])

## DDPM：理论联系最终必须收敛为一个可训练过程

DDPM 把扩散概率过程、变分训练目标和 denoising score matching 联系起来，但最终仍然落到一个明确、可执行的前向加噪与反向生成算法。它说明“数学建模”与“算法”并不完全是同义词，但在经验 AI 研究中，数学建模的主要价值通常需要通过训练或推理程序兑现。([NeurIPS Proceedings][7])

## AlexNet：硬件、实现和算法选择是一个整体

AlexNet 的突破并不只是一条抽象网络结构，还包括大规模卷积网络、非饱和激活、GPU 计算以及正则化等选择的组合。它体现的是算法—数据—硬件的共同设计，而不是把系统实现视为算法完成后的附属工作。([NeurIPS Proceedings][8])

# 对“顶级论文不应该这么抽象”的精确修正

我同意你对当前工作流“间接抽象”的批评，但不同意“不应该抽象”的字面说法。

ResNet、Transformer、ViT、MAML、LoRA 和扩散模型都高度抽象。区别是：

### 好的算法抽象

* 改变明确的数学对象；
* 对应明确的计算图；
* 能写成伪代码；
* 能直接实现；
* 能预测行为；
* 能被 ablation 摧毁；
* 能讨论复杂度与扩展性。

### 不好的元抽象

* 把历史论文总结成宽泛 operator；
* 用 jury 和 Taste anchors 判断“像不像好研究”；
* 为创意生成设计复杂的角色政治；
* 用很多规则间接模拟科学判断；
* 不能说明下一行代码、下一项梯度或下一次实验应该改变什么。

所以，不是“去掉建模”，而是：

> **从研究过程的元建模，回到学习系统本身的算法建模。**

---

# 建议重构成更小的算法闭环

核心工作流可以压缩为：

[
\text{Observe}
\rightarrow
\text{Model}
\rightarrow
\text{Build}
\rightarrow
\text{Run}
\rightarrow
\text{Diagnose}
\rightarrow
\text{Update}
]

## 1. Observe：定位瓶颈

输入必须优先是：

* 代码；
* 数据；
* 训练日志；
* 曲线；
* error slices；
* profiler；
* 现有 baseline；
* 失败案例。

输出只需要一句可证伪判断：

> 当前性能主要受 X 限制，而不是 Y。

## 2. Model：提出算法改动

必须由主算法代理亲自完成。

```text
BOTTLENECK:
ALGORITHM:
- equation or pseudocode
- changed computational path
- changed gradient/information flow

WHY:
PREDICTED SIGNATURES:
SCALING EXPECTATION:
```

外部模型可以提出替代算法，但不能替主代理拥有 working model。

## 3. Build：实现最小忠实版本

这里的“最小”不是玩具化，而是：

> 去掉非必要复杂度后，仍然能够忠实呈现机制的最小实现。

同时记录：

* 修改了哪些模块；
* 参数量和计算量变化；
* 内存与吞吐变化；
* 与 baseline 是否保持公平。

## 4. Run：运行多保真度实验

不应绝对要求“总是先做最便宜实验”。应根据机制选择：

* unit test；
* synthetic case；
* 小规模训练；
* 中等规模 scaling check；
* 完整 claim-bearing run。

某些机制在小尺度下不可见，因此“小实验无收益”不能自动杀死候选。

## 5. Diagnose：比较预测签名与实际签名

重点不是只看总分，而是：

```text
Predicted:
Observed:
Matched:
Contradicted:
Implementation anomalies:
Most likely explanation:
```

## 6. Update：直接修改算法

输出应是：

* 保留机制；
* 修改哪个算法部分；
* 放弃哪个假设；
* 什么新证据会重新打开该方向。

---

# Gate 应缩减到两类

## Spend Gate

只在训练成本、数据成本或不可逆决策达到阈值时触发。

检查：

* 机制是否明确；
* 实现是否通过基本验证；
* 是否存在更低成本但忠实的测试；
* 失败后是否仍有信息价值。

## Claim Gate

只在结果要被写进论文、报告或对外结论时触发。

检查：

* run reality；
* baseline 公平性；
* 统计与多 seed；
* artifact fidelity；
* claim scope；
* 是否存在替代解释。

`exp-verify` 中关于 no-op、artifact、run registry 的内容值得保留，但最好成为机械化工具检查，而不是新的认知流程。

# 具体文件级修改建议

## P0：先改主体

1. 删除 `conductor/SKILL.md` 中 “conductor never authors mathematics”。
2. 删除 `forge/SKILL.md` 中同类 delegated modeling 规定。
3. 改成：主算法代理必须维护当前算法的公式、伪代码、实现映射和行为预测。
4. 外部模型只承担 alternatives、critique、implementation review。

## P0：拆开三个概念

将当前 `taste.md` 拆成：

* `algorithm-taste.md`
* `experiment-policy.md`
* `claim-policy.md`

不要再用一个指标同时控制创意、实验和证据。

## P0：取消 selection jury 的决策权

从 Prospect 删除：

* 强制 cross-family selection jury；
* 强制 GOOD/BAD Taste anchors；
* 强制 program-chair verdict；
* 强制并行 micro-pilot。

可以保留为可选咨询手段，但最终算法选择由主算法代理负责。

## P1：把 Operator Bank 改成 Algorithm Case Memory

每个案例只保存：

```text
Observed failure
Exact intervention
Mechanistic prediction
Observed signature
Boundary condition
```

按当前 bottleneck 检索，不在每次生成前强制加载。

## P1：压缩 MoA routing

MoA 只保留三种可选角色：

* proposer：提出替代算法；
* critic：寻找机制漏洞；
* verifier：检查证据和实现。

不需要 program chair、due process、stance matrix 和复杂 reconciliation protocol。

## P1：采用按需加载

始终在上下文中的内容最好不超过：

* 算法工程师角色定义；
* 当前 Algorithm Card；
* 当前 artifacts；
* 两道 gate 的触发条件。

评审协议、写作规范、统计模板和运行工具在需要时加载，而不是长期占据注意力。

## P2：清理版本漂移

统一：

* README 版本；
* plugin metadata；
* 各 skill version；
* command 和 skill 数量；
* Prospect mine 数量；
* canonical document 的优先级。

## 最后的结论

当前 workflow 最核心的错误，不是某一条 Taste 公式写得不够好，而是：

> **它将完整的 AI 算法工程师拆解成了 Human Taste、外部数学、Conductor 编排、Executor 实现和 Jury 评审。**

结果是每个角色都完成了一部分“科研动作”，却没有任何角色形成完整的算法直觉。

真正需要建模的不是一个越来越复杂的 Taste 评分器，而是一个研究型算法工程师的工作状态：

1. 看 artifact；
2. 定位瓶颈；
3. 亲自定义数学与算法；
4. 理解计算图和实现；
5. 预测可观察行为；
6. 运行忠实实验；
7. 用残差修改算法；
8. 只在提出公共 claim 时引入严格独立审查。

最重要的一句可以作为新系统的总原则：

> **算法所有权不能委派；创意选择不需要程序委员会；只有证据认证必须独立。**
